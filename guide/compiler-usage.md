# 编译器使用

## 命令行用法

```
citius [选项] <输入文件.ct>
```

**基本工作流程：** 指定一个 Citius 源文件（`.ct`），通过选项控制编译的停止阶段和输出形式。

---

## 选项详解

| 选项 | 作用 | 编译停止阶段 | 输出产物 |
|------|------|-------------|----------|
| `-o <文件>` | 指定输出文件名 | 生成可执行文件 | 可执行文件 |
| `-S` | 仅生成 C 代码，不编译 | CodeGen 之后 | C 源码（stdout） |
| `-c` | 编译到目标文件 | 系统 C 编译器之后 | `.o` / `.obj` |
| `-O<级别>` | 优化级别控制 | 传给系统 C 编译器 | 影响生成效率 |
| `-run` | 编译并立即运行 | 编译完成立即执行 | 临时文件（自动清理） |
| `-help` | 显示帮助信息 | 解析参数后 | 帮助文本 |

### -o \<文件\>

指定最终可执行文件的输出路径：

```bash
# 编译 hello.ct，生成可执行文件 my_program（Windows 上为 my_program.exe）
citius hello.ct -o my_program
```

**编译流程：** `hello.ct` → CodeGen → `hello.c` → 系统 C 编译器 → `my_program`（或 `my_program.exe`）

### -S —— 仅查看 C 代码

`-S` 让编译器在**代码生成阶段**后停止，将生成的 C99 源码输出到标准输出。这是学习和调试 Citius 最重要的选项。

```bash
# 查看 hello.ct 生成的 C 源码
citius hello.ct -S
```

**典型输出示例：**

```c
#include <stdio.h>
#include <stdint.h>
#include <stdbool.h>

static int main(void) {
    printf("%s\n", "Hello, Citius!\n");
    return 0;
}

// 如果源码中包含 sleep 语句，还会自动生成：
// static void citius_sleep(int64_t ms) { ... }
// 如果源码中包含 sent 语句，还会自动生成：
// static void citius_http_request(...) { ... }
```

**用途：**
- 确认你的 Citius 代码被正确翻译
- 学习 Citius 的特性是如何映射到 C 语言的
- 手动检查编译器输出是否包含预期的辅助函数

### -c —— 编译到目标文件

只编译到目标文件（`.o` / `.obj`），不进行链接。适合大型项目分模块编译，最后手动链接：

```bash
# 生成 hello.o 或 hello.obj
citius hello.ct -c
```

### -O\<级别\> —— 优化级别

控制传递给系统 C 编译器的优化参数，从 `-O0`（无优化）到 `-O3`（最大优化）。默认值为 `-O2`。

```bash
# -O0：不优化，编译最快，适合调试
citius hello.ct -O0 -o hello_debug

# -O3：最大优化，运行最快，编译较慢
citius hello.ct -O3 -o hello_fast
```

| 级别 | 编译器标志 | 编译速度 | 运行速度 | 适用场景 |
|------|-----------|----------|----------|----------|
| `-O0` | `-O0` | 最快 | 最慢 | 调试、开发 |
| `-O1` | `-O1` | 较快 | 较慢 | 日常开发 |
| `-O2` | `-O2` | 中等 | 快（默认） | 发布版本 |
| `-O3` | `-O3` | 最慢 | 最快 | 性能关键场景 |

### -run —— 编译并立即执行

一步完成编译、运行、清理，适用于快速测试：

```bash
citius hello.ct -run
```

**内部流程：**
1. 编译 `hello.ct` → 生成临时 C 文件（如 `_citius_tmp_1234.c`）
2. 调用系统 C 编译器 → 生成临时可执行文件
3. 执行临时可执行文件
4. 程序退出后，删除所有临时文件

### -help —— 查看帮助

```bash
citius -help
```

显示所有可用选项的简要说明。

## 完整示例

### 1. 开发调试流程

```bash
# 第 1 步：查看生成的 C 代码，确认逻辑正确
citius my_program.ct -S

# 第 2 步：编译为可执行文件（默认优化 -O2）
citius my_program.ct -o my_program

# 第 3 步：运行测试
./my_program
```

### 2. 性能调优流程

```bash
# 先用 -O3 编译发布版本
citius my_program.ct -O3 -o my_program_release

# 用 -O0 保留一个调试版本
citius my_program.ct -O0 -o my_program_debug
```

### 3. 模块化编译

```bash
# 分别编译各个模块为目标文件
citius module1.ct -c   # 生成 module1.o
citius module2.ct -c   # 生成 module2.o

# 手动链接（使用系统 C 编译器）
gcc module1.o module2.o -o program
```

## 编译器架构详解

Citius 编译器采用经典的四阶段设计，每个阶段职责明确：

```
源文件 (.ct)
    │
    ▼
┌─────────────────────────────────────────────┐
│ 1. Lexer (词法分析)                          │
│    ┌─────────────────────────────────────┐   │
│    │ 输入: 字符序列 "func main() { ... }" │   │
│    │ 输出: Token 序列                      │   │
│    │ Token: [KEYWORD(func), ID(main),     │   │
│    │         LPAREN, RPAREN, LBRACE, ...] │   │
│    └─────────────────────────────────────┘   │
│    源码位置: src/Lexer/Lexer.cpp              │
└─────────────────────┬───────────────────────┘
                      ▼
┌─────────────────────────────────────────────┐
│ 2. Parser (语法分析)                         │
│    ┌─────────────────────────────────────┐   │
│    │ 输入: Token 序列                      │   │
│    │ 算法: 递归下降 + 优先级爬升           │   │
│    │ 输出: 抽象语法树 (AST)                │   │
│    │ AST: Program → Function(main)        │   │
│    │        → Body → CallExpr(print)      │   │
│    └─────────────────────────────────────┘   │
│    源码位置: src/Parser/Parser.cpp            │
└─────────────────────┬───────────────────────┘
                      ▼
┌─────────────────────────────────────────────┐
│ 3. Sema (语义分析)                           │
│    ┌─────────────────────────────────────┐   │
│    │ 输入: AST                             │   │
│    │ Pass 1: 收集所有函数声明              │   │
│    │ Pass 2: 类型检查 + 符号解析 + 推断    │   │
│    │ Pass 3: 重新分析推断出的参数类型      │   │
│    │ 输出: 类型标注完成的 AST              │   │
│    └─────────────────────────────────────┘   │
│    源码位置: src/Sema/Sema.cpp                │
└─────────────────────┬───────────────────────┘
                      ▼
┌─────────────────────────────────────────────┐
│ 4. CodeGen (代码生成)                        │
│    ┌─────────────────────────────────────┐   │
│    │ 输入: 类型标注完成的 AST              │   │
│    │ 输出: C99 源码                        │   │
│    │ 额外生成:                              │   │
│    │  • #include 头文件                    │   │
│    │  • citius_sleep() (检测到 sleep 时)  │   │
│    │  • citius_http_request() (检测到 sent) │   │
│    │  • 所有函数前向声明                   │   │
│    └─────────────────────────────────────┘   │
│    源码位置: src/CodeGen/CodeGen.cpp          │
└─────────────────────┬───────────────────────┘
                      ▼
               生成的 C 代码 (.c)
                      │
                      ▼
         系统 C 编译器 (Clang/GCC/MSVC)
                      │
                      ▼
               可执行文件 (.exe)
```

## C 编译器选择逻辑

CodeGen 完成后，Citius 会在系统上按以下顺序查找可用 C 编译器：

| 优先级 | 编译器 | 检测方式 | 平台 |
|--------|--------|----------|------|
| 1（最高） | Clang | 查找 `clang` 命令 | 全平台 |
| 2 | GCC | 查找 `gcc` 命令 | 全平台 |
| 3 | MSVC | 查找 `cl.exe` | Windows |

找到第一个可用编译器后，使用它编译生成的 C 代码。链接时自动添加：
- **Windows:** `-lws2_32`（Winsock 网络库，仅在使用了 `sent` 语句时）
- **POSIX:** `-lm`（数学库）

## 调试技巧

### 查看中间产物

使用 `-S` 选项是调试和理解 Citius 程序的最佳方式：

```bash
# 将生成的 C 代码保存到文件，便于反复查看
citius my_program.ct -S > output.c

# 然后用任何文本编辑器或 IDE 查看 output.c
```

### 常见调试场景

| 场景 | 做法 | 预期结果 |
|------|------|----------|
| 想了解 print 如何编译 | `citius test.ct -S` | 看到 `printf` 调用 |
| 想验证 sleep 是否工作 | 包含 `sleep` 语句后 `-S` | 看到 `citius_sleep()` 辅助函数 |
| 想检查 sent 的代码 | 包含 `sent` 语句后 `-S` | 看到 socket 相关代码 |
| 编译器报语法错误 | 检查最近的修改 | 仔细查看行号和错误信息 |
| 类型推断不符合预期 | 检查参数和返回值类型 | 给关键参数加上显式类型注解 |
